LEARNING NOTE

The = vs += bug is a classic "replace vs accumulate" mistake — appears when you write a function that handles both "first add" and "update" in one branch. LEARNING NOTE

VAD (Voice Activity Detection) is the mechanism that detects when a speaker has stopped talking. Gemini has built-in VAD (~800ms–1s timeout) but it only fires if you never tell it explicitly that speech ended. send_audio_stream_end() is a signal that says "I'm done talking, you can reply now." This is the correct production pattern — the application knows speech ended (user released the button) before any VAD model does. Always use explicit signals over timeouts when you have them.

LEARNING NOTE

Sequence vs simultaneity is a core UI animation concept. When multiple things happen at once they compete for attention and look chaotic. When they're sequenced (A then B then C) the user's eye is guided — they see A, then B, then C. The server controls the intent (what should happen); the widget controls the execution (when and how fast). Separating these concerns (intent in Python, timing in JS) keeps your Python clean while letting the UI be as polished as needed.

WHAT TO CHANGE — The full sequence behavior

When search_products returns N results, the dispatcher sends N HighlightProduct actions. Each action now carries:

Phase 4 — Search Quality

The search is the product. Currently it works but has dead code, a loose threshold, and no way to get "more results" or filter by real category. This phase makes search feel genuinely smart.

4A
Unify Embedding Text + Quality Fixes
Single builder · Better semantic content · Tighter threshold
4A
Delete dead code, fix entities, include short_desc, raise threshold to 0.5
DO NOW ~1.5h
WHAT TO CHANGE
  • Single embedding builder: Product.to_embedding_text() in models/product.py is defined but never called — rag_service._build_rich_text() is what actually runs. Delete _build_rich_text() and make rag_service call product.to_embedding_text(). One place, one version, no drift.

  • Decode HTML entities: In to_embedding_text(), run html.unescape(description) before the regex strip. Fixes &ndash; → —, &amp; → &, etc. These literal entity strings corrupt the embedding's semantic meaning.
  • Raise threshold: vector_store.py line 151: change score_threshold=0.3 to 0.45. On a 250-product catalog, 0.3 is too loose and returns semantically unrelated products.

FILES TO CHANGE
src/voice_commerce/models/product.py to_embedding_text(): add short_desc, html.unescape
src/voice_commerce/services/rag_service.py replace _build_rich_text() call with product.to_embedding_text()

LEARNING NOTE

"Garbage in, garbage out" is especially true for embeddings. The embedding model encodes text into a 384-dimensional vector that represents the text's meaning in geometric space. If your text says &ndash; instead of —, the model sees a weird string instead of a dash. When a user searches "wireless noise–canceling headphones," the query embedding and the product embedding won't align well because the product text had corrupted punctuation. The fix is cheap; the impact on search quality is significant.

Pagination via offset is standard in every search API (Google: start=10, Elasticsearch: from=10, Qdrant: offset=10). The pattern: you return N results starting from position offset. On "show me more", increment offset by N. The per-session cache is an extra guard: even if Qdrant returns slightly different results on a repeat query (due to ANN approximation), you've already filtered out seen IDs. This makes the user experience smooth even if the underlying similarity scores fluctuate.

5A
Passive Event Tracking (Agent Closed)
Agent knows what user did before opening it
5A
Queue lightweight user events when agent is closed → flush on open/mic/text
DO NOW ~2h
Your new point: When the voice agent is closed, the user may be browsing: applying filters, sorting products, navigating to a new page. We don't send full page context (no WebSocket connection or agent is paused), but we can queue small event lines in memory. When the agent opens (or user starts mic/types), we send those queued events + the current page state. The agent then knows "the user filtered to Men's, sorted by price low-to-high, then opened you."
WHAT TO CHANGE
  • widget.js — event queue: Add STATE.eventQueue = []. Listen for DOM mutations/events on filter buttons, sort dropdowns, URL changes (popstate), price range inputs — even when the panel is closed. On each event, push a one-line string: "User filtered: category=Men", "User sorted: price asc", "User navigated: /shop/hoodies", "User set price range: $50–$300". Max queue: last 10 events.
  • Flush on connection: When WebSocket connects (on panel open, mic start, or first text), include STATE.eventQueue in the first context message, then clear the queue.
  • Also flush immediately when user starts mic or sends text: Before sending audio/text, check if queue is non-empty → send as a context_update message first, then the audio/text. This ensures Gemini has the context before the user's first word.
  • Backend — new context field: voice_websocket_handler.py receives context_update with a new events list. Pass to gemini.inject_live_context(events=events, ...) and include in the formatted context message.
WHY THIS MATTERS

Without this, the agent has no idea what the user was doing before opening the widget. The user might have spent 2 minutes browsing hoodies, applied a price filter, and then asks "what do you think?" — and the agent starts from scratch. With the event queue, Gemini knows exactly where the user was in their journey. This makes the agent feel contextually aware instead of amnesiac.

FILES TO CHANGE
static/widget.js STATE.eventQueue, DOM event listeners (even when panel is closed), flush logic on connect/mic/text
static/embed_demo.html emit filter/sort/navigation DOM events that widget.js can intercept
src/voice_commerce/handlers/voice_websocket_handler.py handle events field in context_update message
src/voice_commerce/core/voice/gemini_live_handler.py inject_live_context: include events list in formatted message
LEARNING NOTE

This is the "event sourcing" pattern — instead of storing state (full page snapshot), you store what happened (event log). The event log is lightweight (10 text strings), interpretable (Gemini reads English), and gapless (even if the page changed 5 times while the agent was closed, you have a record). Full page state is complementary — you send both: "here's what happened" + "here's what it looks like now." One answers "how did we get here?" the other answers "what do we see now?"
